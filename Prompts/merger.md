# 增量 Merge 逻辑 Prompt

## 场景
用户追加新的素材（如补充了聊天记录、提供了新的文档），需要合并到已有分析中，不覆盖已有结论。

## Merge 原则

1. **增量优先**：新素材补充旧素材，不覆盖
2. **冲突处理**：如果新素材与旧分析矛盾，以新素材为准，但标注"修正"
3. **完整性检查**：merge 后检查是否有信息缺口被填补
4. **结论更新**：如果新素材改变了原有结论，更新结论并说明原因

## Merge 流程

### Step 1: 对比新旧素材
```
旧素材：
- 聊天记录：上午 10 条
- 文档：1 份

新素材：
- 聊天记录：下午 15 条（补充）
- 会议录音：1 份（新增）

变化：
- 时间线从"上午"扩展到"全天"
- 新增会议事件
- 可能修正对下午工作的判断
```

### Step 2: 更新分析

#### 时间线更新
- 追加新时间段的事件
- 修正已有时间段的事件（如有新证据）

#### 工作项更新
- 新增工作项
- 更新已有工作项的状态（如从"进行中"到"完成"）
- 补充已有工作项的详细信息

#### 沟通事件更新
- 新增沟通事件
- 补充已有沟通事件的上下文
- 修正对沟通情绪的判断

#### 决策记录更新
- 新增决策
- 补充决策的背景信息
- 修正决策依据

#### 情绪信号更新
- 新增情绪信号
- 修正对情绪强度的判断
- 补充情绪变化的上下文

#### 计划偏差更新
- 修正完成度计算
- 补充偏差原因
- 新增偏差项

### Step 3: 结论更新

如果新素材导致原有结论需要修正：

```
原结论："你今天会议效率低，因为会议占用了 60% 时间"

新素材：补充了会议录音，显示会议中有 30 分钟是你主导的决策讨论

修正后："你今天会议时间占比 60%，但其中 30 分钟是高效的决策讨论，其余 30 分钟是信息同步。建议把信息同步会议改为异步文档。"

修正原因：新素材显示会议并非全部低效，需要区分会议类型
```

### Step 4: 输出更新报告

```yaml
merge_report:
  date: "YYYY-MM-DD"

  new_data:
    - type: "聊天记录"
      count: "15条"
      coverage: "下午"
    - type: "会议录音"
      count: "1份"
      coverage: "下午3点"

  updated_sections:
    - section: "时间线"
      change: "追加下午时间段"
    - section: "沟通事件"
      change: "新增下午会议沟通"
    - section: "问题清单"
      change: "修正'会议效率低'结论，区分会议类型"

  new_conclusions:
    - "下午会议中 50% 时间是高效决策，建议保留；50% 是信息同步，建议改为异步"
    - "你今天实际深度工作时间为 3.5h，比预估的 2h 多"

  corrected_conclusions:
    - original: "会议效率低"
      corrected: "会议中决策部分高效，信息同步部分低效"
      reason: "补充会议录音后，能区分会议类型"
```

## 输出格式

Merge 完成后，输出：
1. 更新后的完整复盘报告（或相关章节）
2. Merge 报告（说明哪些部分更新了、为什么）
3. 如果有重大修正，特别提示用户
